# ARES Lite Refactor Plan (Async + Streaming + Resilience)

## Goals
- **Non-blocking runs**: `POST /api/run` must return quickly with a `run_id` and queue work in the background.
- **Offline/local-first**: no external infra required; default worker runs inside the backend process.
- **Memory efficiency**: do not hold all video frames in RAM at once.
- **Resilience**: clear errors for missing `ffmpeg/ffprobe`, invalid annotations, invalid detector outputs; avoid crashes.
- **Stable behavior**: preserve current metrics/readiness semantics where possible; when behavior must change, expose it explicitly.

## High-Level Architecture

### Data Model (SQLite)
`runs` table is extended with run lifecycle fields:
- `status`: `queued | processing | completed | failed`
- `stage`: fine-grained progress stage (see below)
- `progress`: integer percent `0..100`
- `message`: short human-friendly progress detail
- `error_message`: set on failure
- timestamps: `queued_at | started_at | finished_at | updated_at`

We keep SQLite, but enable:
- WAL journal mode
- busy timeout
- (optional) reduced sync mode for dev stability

### Job Queue / Worker
- **SQLite-backed queue**: the `runs` table is the source of truth.
- A **single worker thread** polls the DB and atomically claims queued runs.
- On startup:
  - stale `processing` runs are re-queued or failed (crash recovery)
- Worker executes: `extract -> stress -> inference -> metrics -> report` and updates `runs.stage/progress`.

### Pipeline (Streaming Frames)
Old behavior loaded all extracted frames into RAM. New behavior:
1. Extract sampled frames to disk using `ffmpeg` (same as before).
2. Iterate frames **one-by-one**:
   - `cv2.imread` one frame
   - apply stressors for that frame
   - write stressed frame to disk
   - track dropped frames (frame_drop) without storing image arrays
3. Run inference over the **list of stressed frame paths** (paths only; no images kept in RAM).
4. Persist detections in **batches** to reduce SQLite locking.

Overlays are generated by reloading a single stressed frame from disk.

### Box Format & Validation
Internal canonical bbox format is:
- `bbox: [x, y, w, h]` (xywh), integers, `w>=1`, `h>=1`

We validate + normalize:
- Ground truth boxes loaded from `backend/data/annotations/*.json`
- Predicted boxes emitted by detectors

Invalid box schemas fail the run with a clear `error_message` and do not crash the server.

### Baseline Redesign
We introduce a stable `baseline_key` stored in `runs.config_json` for completed runs:
- `baseline_key = "{video_id}|{detector_backend}|{resize}|{every_n_frames}|{max_frames}"`

Baseline matching is now:
- find most recent completed run with the same `baseline_key` AND `stress_enabled=false`

If no baseline exists:
- metrics include `baseline_missing: true`
- readiness does **not** force a synthetic robustness score; robustness becomes `null` and readiness is computed with **renormalized weights** (no penalty, but explicitly "missing evidence").

### Reports ("latest" behavior)
- Reports remain per-run under `backend/data/runs/<run_id>/index.html`.
- We **do not** overwrite a global `results/latest/report/index.html`.
- Optionally we write an atomic pointer file:
  - `backend/data/runs/latest.json` containing `{ run_id, timestamp }`

## API Changes

### POST /api/run (async)
Creates a run row (status `queued`), enqueues work, returns immediately.

Response remains shape-compatible with the old `RunResponse`:
- `status` is now `queued` or `processing` initially
- numeric fields (`frames_processed`, `inference_seconds`, etc.) are `0` until completion

### GET /api/runs/{run_id}
Returns:
- existing fields: `id`, `scenario_id`, `status`, `created_at`, `config`
- new fields: `stage`, `progress`, `message`, `error_message`, timestamps

### Existing endpoints
Metrics / engagement / readiness / blindspots remain unchanged, but will 404 until the run completes.

## Progress Stages
Stages are stable strings:
- `queued`
- `extracting_frames`
- `stressing_frames`
- `inference`
- `persisting`
- `metrics`
- `reporting`
- `completed`
- `failed`
- `cancelled`

## Developer Notes
- Default worker concurrency is 1 to reduce SQLite lock contention.
- We avoid heavy deps. No Celery/Redis.
